1.1介绍

在这一章中，我们将 在Mathematica 中介绍函数的概念，并考虑许多函数的例子。由于Mathematica编程语言在很大程度上是一种函数式编程语言，因此函数是本文的中心对象。此外，由于列表被用作数据结构的通用构建块，任何复杂的数据结构都可以用列表构建并“动态地”修改，因此与面向对象编程语言相比，重点更多地转向了函数。

Mathematica中函数和函数式编程的另一个重要方面是，这个编程的“层”是建立在一个(在我看来，Mathematica中更基本的)基于规则的引擎之上的。这就产生了函数的概念，它比大多数其他编程语言更广泛，也有很大的不同。因此，如果不了解规则和基于规则的编程技巧，就不可能完全掌握Mathematica中的函数。我们也将在这里讨论这一话题。

1.2Rule（规则）和patten（模式）

为了更好地理解Mathematica中的函数，我们需要对模式和规则替换有很好的理解。这两个主题仅仅是一个主题的两个方面，因为模式的形式决定了关联规则作用在哪个对象类上。

1.2.1.2 应用规则 ：用ReplaceAll函数

我们定义的任何规则本身都不能做任何有用的事情。

命令规则本身只是规则左右两边的命名容器。当与另一个命令结合使用时，它就变得非常有用了， 该命令在表达式中执行规则替换。

这个命令有一个格式 ReplaceAll[expr, rules] 和一个简写符号:expr/.rules。

当我们我们应用多个规则，它们通常应该放在一个简单的(而不是嵌套的)列表中，稍后我们将看到这样的例子。

f[a] /. rule

规则应用程序通常从较大的表达式开始。

Clear[a, q];

q[{{a}}]

上面的代码看起来似乎不是很清楚， 我来解释一下，首先看这个x_，这个x先不要管。你先看这个_符号，这个符号_的右边没有东西就表示头部是任意表达式的表达式。这里的x只是一个符号可以是任何mma表达式，那么x_代表的意思就是：头部为任意表达式的表达式。然后{x_}就代表{任意表达式}，那么替换{x_}:>q[x]就代表{一些东西}替换成q[一些东西]，在这里的一些东西是{{a}}，所以最后输出为q[{{a}]]

1.2.1.4规则的结合律

正如前面的例子所说的，规则的应用是左关联的，这意味着表达式 expr /.规则1 /.规则2 是合法的，首先将规则1应用于 expr ，然后将规则2应用于得到的结果 .

1.2.1.5规则的位置

需要说明的是，像上面这样的规则是局部的。

这意味着，当规则重写其应用于其他对象的对象时，它将更改该对象的副本，而初始对象保持不变。在上面的例子中，表达式f[a]没有变化 。

这是规则和执行类似转换的函数之间的主要区别。

在后一种情况中，类似的规则是全局定义的(这意味着内核将自动尝试以交互方式输入的任何表达式，或者以其他方式计算)。

本质上，这是Mathematica中函数和规则列表之间唯一的基本区别。

例如，我们可以通过局部规则轻松地模拟平方函数 

Clear[f];
{f[x], f[y], f[something], f[10]} /. f[x_] :> x^2
{x^2, y^2, something^2, 100}

1.2.1.6用RuleDelayed函数自定义规则

我用这个例子介绍了两个新概念。

第一，我们在规则的左侧有一个模式——它们用于扩大规则，将应用到相应的对象。

第二，我们使用了一种新的规则(只有两个规则，其中一个我们以前已经考虑过了)，即使用:>符号代替->符号。它的字面等价是RuleDelayed[lhs,rhs]命令 

RuleDelayed[a, b]
a :> b

我们通过名称可以猜到的，这对应于一个延迟的规则替换。

也就是说，规则的r.h.s.（right hand side赋值操作的右侧）（说白了就是等式的右侧）是在规则替换发生之后才计算，也就是说先替换，再计算！。

稍后我们将详细讨论规则或RuleDelayed更合适的情况，但是一般情况下，这里的情况与使用Set和SetDelayed赋值操作符的情况类似。这种相似性并不是偶然的，而是再次反映了Mathematica中的赋值运算符只是创建全局规则的运算符这一事实。

1.2.1.7Rule(规则)和RuleDelayed（延迟规则）的区别举例。

为了举例说明Rule和RuleDelayed在一个特定示例中的区别。

考虑这样的一个问题:给定一个元素列表，将符号 a 的每次出现都替换为一个随机数。

Clear[sample, a, b, c, d, e, f, g];
sample = {a, b, d, e, g, f, c, b, a, d}
(*下面我们分别用\[Rule]和\[RuleDelayed]操作*)
sample /. a -> Random[]
sample /. a :> Random[]
{0.741878, b, d, e, g, f, c, b, 0.741878, d}
{0.14384, b, d, e, g, f, c, b, 0.160757, d}

在线运行

我们看到

第一种情况下的数字是相同的。

而第二种情况下的数字是不同的。

这是因为

在第一种情况下，规则的r.h.s（右边）.在应用之前已经被计算过了，而且是一次性的。

在第二种情况下，每次应用规则时都要重新评估规则的r.h.s.。

下面我再给一个例子

1.2.2规则替换没有交换律

1.2.2.1 规则的列表

当我们在一个表达式上使用多个规则时，我们将这些规则放到一个列表中。例如 

{a, b, c, d} /. {a -> 1, b -> 2, c -> 3}

{1, 2, 3, d}

在线运行

对于要在表达式上使用的所有规则，规则列表必须是平面列表（类似与这种{something}，这种就不行{{something1},{something2}}），也就是说，所有列表元素都应该有一个head规则或RuleDelayed。

嵌套的规则列表来替换或ReplaceAll并没有错错误，而是被解释为我们想要在原始表达式的几个副本上分别尝试规则的所有子列表 。

举个例子来说明一下嵌套列表：

{a, b, c, d} /. {{a -> 1, b -> 2, c -> 2}, {a -> 1}, {c -> 2, d -> 3}}
{{1, 2, 2, d}, {1, b, c, d}, {a, b, 2, 3}}

在线运行

1.2.2.2 没有交换律

规则替换的结果通常取决于规则在列表中的顺序，如下面的示例所示。 

Clear[a, f];
f[a] /. {a -> 5, a -> 7}
f[a] /. {a -> 1, a -> 4}
f[5]
f[1]

在线运行

mma这样做的原因是，一旦某个规则被应用到表达式的给定部分，ReplaceAll就会转到表达式的下一部分，并尝试下一部分的规则。

但是，即使我们多次运行ReplaceAll，对于列表中规则的不同排序，结果通常仍然是不同的。这是因为一旦一个规则应用到一个(部分)表达式上，这个部分通常会被重写，这样我们之前应用的规则列表中的一些规则将不再适用，反之亦然。

在任何情况下，我们的最终结论是规则应用程序不是可交换的，规则列表中的规则顺序通常很重要。对于这方面的一个极端示例，我们将很快考虑一个基于规则的阶乘函数，其中不同的规则排序将导致无限迭代

1.2.3 规则和计算过程之间的相互作用

在Mathematica中，需要注意的是我们从来没有真正从Head（头部）开始，而是始终使用一个大型的内置规则系统，将内置函数连接在一起。

这为使用这些函数提供了很大的灵活性，因为可以使用用户定义的规则操作或覆盖这些系统规则。

另一方面，必须小心的是，因为用户新定义的规则(或函数定义和变量赋值，它们是全局规则)立即开始与内置规则交互。

1.2.3.1 当规则被应用时，表达式被计算

作为第一个例子我们考虑伽马函数：

Clear[a];
Gamma[a]
Gamma[a] /. a -> 2
Gamma[a]
1

在线运行

在上面的例子中由于系统不知道a是什么，所以与gamma函数相关的任何规则都不适用，只是返回输入。

当我们用规则替换a->2的时候， 我们可以看到，一旦数值(整数)被替换，就会应用一个内建规则来产生结果。

与此同时,对于派这个数，没有规则迫使Mathematica计算数值,因此不会计算出来。

Gamma[a] /. a -> Pi
Gamma[\[Pi]]

在线运行

如果我们想在这种情况下计算这个数值，我们可以这样做 

Gamma[a] /. a -> N[Pi]
2.28804

或者这样也可以

N[Gamma[a] /. a -> Pi]
2.28804

事实上，在某些情况下，我可能希望保持Gamma[5]函数的未求值形式。更一般地说，使用基于规则的方法的优点是，我们不需要为我们想要描述的新情况添加规则。

这并不是说Mathematica的计算是不可预测的，而是说我们编写的程序不应该依赖于那些纯粹由惯例定义的特性。

特别的，在Mathematica中，我们应该总是假设所有的表达式都可以求值为其他的东西。因此，如果某个表达式必须在一段时间内保持未求值，那么程序员必须处理它。另一方面，如果某个表达式必须被完全求值(例如，一个数值)，程序员必须再次确保它。

1.2.3.2 规则的适用性影响的评估

考虑这样的一个例子

{f[Pi], Sin[Pi], Pi^2}
{{f[Pi], Sin[Pi], Pi^2}} /. Pi -> a
{f[\[Pi]], 0, \[Pi]^2}
{{f[a], 0, a^2}}

在线运行

在上面的代码的第二个，当我们应用Pi->a这个规则的时候，我们会很自然的认为应该是以Sin[a]出现，而不是以0输出。

为什么会这样？

我们曾经说过/.只是ReplaceAll的缩写！，我们把这个完整的写出来。

RepalceAll[{f[Pi], Sin[Pi], Pi^2}, Pi -> a]
{{f[a], 0, a^2}}

现在我们回想一下Mathematica中的计算法则，子表达式通常在表达式之前进行计算。

这意味着当计算过程到达ReplaceAll命令的时候，我们的表达式就已经转换为与第一个输入的输出相同的形式(没有规则替换之前的形式)。

在没有用ReplaceAll命令之前，Sin[Pi]的值为0，由于0不再包含Pi，因此不符合规则，所以表达式的这一部分没有进一步的替换。

另外，我们可以通过使用Trace命令查看计算过程

Trace[{{f[Pi], Sin[Pi], Pi^2}} /. Pi -> a]
{{{{Sin[\[Pi]],0},{f[\[Pi]],0,\[Pi]^2}},{{f[\[Pi]],0,\[Pi]^2}}},{\[Pi]->a,\[Pi]->a},{{f[\[Pi]],0,\[Pi]^2}}/. \[Pi]->a,{{f[a],0,a^2}}}

在线运行

这些示例可能会给人一种印象，即Mathematica在与规则排序相关的方面是有bug的。

虽然Mathematica程序中的许多重要bug确实与这个问题有关，但也有一些方法可以避免它们。

只要规则或规则列表总是正确的，它们要么表示精确的恒等式(例如，数学恒等式)，要么可以确保在没有实际情况下，单独使用它们不会导致错误的结果，就应该没有问题。

当被认为“正确”的规则在某些不可预见的情况下给出了不正确的结果时，bug就会发生，但是对于在更传统的编程范例中编写的程序也是如此。

也许，真正的区别在于，对于更传统的编程技术，通常更容易将程序限制在问题参数空间的“角落”中，在那里可以预测或证明正确的性能。

我个人认为，规则排序所引起的复杂性是(可能不可避免的)为使用一种非常普遍的评估方法而付出的代价。

模式标记不能是复合表达式，只能是真正的符号(Head为symbol的)。

模式标记的存在并不会以任何方式改变匹配，它只会给我们提供额外的信息。我们不会通过MatchQ获得这个信息，因为MatchQ只是建立了匹配的事实。为此，我们需要一个真正的规则应用程序，因为规则将告诉我们如何处理这些匹配(子)表达式。例如，这里我们将简单地将它们收集到一个列表中

总结:

每当一个模式包含一个部分是一个特殊的符号像空白[]，可能附有标签模式,这意味着实际的表达式匹配可以包含在这个地方,而任意子表达式(任意,取决于所使用的特定的特殊符号)。然而，不包含这些符号的部分，必须以完全相同的方式出现在模式和表达式中，以使它们匹配。

1.2.4.5 有两个参数的函数，但第一个参数是固定的

首先，让我们构建一个规则，它将对具有两个参数的元素起作用

{x, Sin[x], x^2, x*y, x + y, g[y, x], h[x, y, z], 
  Cos[y]} /. {f_[x, z_] -> f[10, z]}

{x, Sin[x], 100, 10 y, 10 + y, g[y, x], h[x, y, z], Cos[y]}

Try it online!

我们看到一个新的模式是f_[x, z_]，它表示两个参数的任意函数，第二个参数是任意的，但是第一个参数固定为x 。

注意，没有对g[y,x]进行替换，因为这里虽然有两个参数，但是第一个参数不是x。

一般来说,对于简单的模式,来确定是否一个给定的模式将匹配一个给定的表达式是考虑的FullForm模式和表达,和模式表达式是否有空白的地方它不同于表达式匹配。

上面模式的完整形式是 ：

f_[x, z_] // FullForm

Pattern[f,Blank[]][x,Pattern[z,Blank[]]]

为了更好地理解上面的模式是如何匹配我们列表中的表达式的，我们可以构造一个不同的规则，它将告诉我们匹配了哪些部分 ：

{x, Sin[x], x^2, x*y, x + y, g[y, x], h[x, y, z], 
  Cos[y]} /. {f_[x, z_] -> {{"f is", f}, {"z is", z}}}

{x,Sin[x],{{f is,Power},{z is,2}},{{f is,Times},{z is,y}},{{f is,Plus},{z is,y}},g[y,x],h[x,y,z],Cos[y]}

{x, Sin[x], x^2, x*y, x + y, g[y, x], h[x, y, z], 
  Cos[y]} /. {f_[x, z_] -> f[10, z], f_[z_, x] -> f[z, 10]}

{x, Sin[x], 100, 10 y, 10 + y, g[y, 10], h[x, y, z], Cos[y]}

Try it online!

1.2.4.7新模式——建立块，空白序列，和一个小警告。

现在让我介绍另一个模式—类似于Blank[]的构建块，它简化了许多模式的构建。这个新模式对象的文字命令是BlankSequence[]。它的字面等价是一个双下划线符号_ _

BlankSequence[ ]

__

Try it online!

此模式元素用于表示表达式中一个或多个元素的序列。例如

{g[x, y, z], g[q, m], h[x]} /. g[t__] :> {t}

{{x, y, z}, {q, m}, h[x]}

Try it online!

由于这种模式构建块允许创建匹配更广泛的表达式类的模式，因此有时我们可能会得到我们不希望得到的结果。

例如，在这里，我们希望向列表中的所有函数添加一个a作为额外的参数 :

{g[x, y, z], g[q, m], h[x]} /. f_[t__] :> f[t, a]

{g[x, y, z], g[q, m], h[x], a}

Try it online!

这里的a被加到列表中去了，不是我们想要的，为什么会这样？。我们用FullForm函数看清它的本质。

{g[x, y, z], g[q, m], h[x]} // FullForm

List[g[x,y,z],g[q,m],h[x]]

我们看到head（头部）为列表匹配f_ (表示任意head（头部）)， t__匹配于列表的内部，因为它确实是一个由三个元素组成的序列—— g 和 g ，h。

因此，a 被作为最后一个参数添加到我们的函数列表中，而不是添加到列表中的每个函数中。我们看到我们的规则比我们想要的高1级,应用于整个表达式而不是部分表达式。

这是使用ReplaceAll时的常见情况。在这种情况下，规则首先在较大的表达式上尝试，如果它们匹配，规则将应用于它们，而不再尝试子表达式。

有几种方法可以改变这种行为。

一种方法是创建受限制的模式，使它们更具选择性。

另一种方法是使用Replace命令，它以不同的顺序执行规则替换

1.2.4.8 有条件限制的模式

如果选择第一种方法，则必须修改模式，使头列表不匹配。我们想要做的是强加一个条件，即f 中的模式 f 与List不相同。

{g[x, y, z], h[x, y]} /. f_[t__] /; f =!= List -> f[t, a]

{g[x, y, z, a], h[x, y, a]}

Try it online!

1.2.4.9 使用Replace代替ReplaceAll

如果我们选择使用Replace命令，这里有一种方法可以做到这一点

Replace[{g[x, y, z], h[x, y]}, f_[t__] :> f[t, a], 1]

{g[x, y, z, a], h[x, y, a]}

Try it online!

Replace和ReplaceAll的区别在于：

Replace允许我们指定必须应用规则的表达式级别（“层”）。这样可以更详细地控制规则的应用。

例如，在这里，我们要求规则只应用于级别1，这意味着列表子表达式的级别，函数g和 h

另一个方法是,我们可以表明该规则适用于整个表达式,我们可以通过使用Replace[expr,rules,Infinity]的顺序规则将应用不同ReplaceAll:现在的规则将应用在深度-第一个方式,从最小的(内部)大(外部)的子表达式。这种行为通常是首选的，然后必须使用Replace。

1.2.4.10 Rule中的规则，一个更好的解决方案

考虑我们最初的问题， 有这么多规则来描述一般情况的一些特殊情况是相当不方便的:每当 x 是某个函数的一个参数时，在该函数中将其更改为< 10 >。

让我们看看能否写出这个问题的更简明、更一般的解。我建议的方法需要BlankSequence。

说明表达结构：

为了说明它是如何工作的，我们现在将创建一个规则，根据这个模式“解构”我们的表达式

{x, Sin[x], x^2, x*y, x + y, g[y, x], h[x, y, z], Cos[y]} /. 
 f_[t__] /; f =!= List -> {{"function", f}, {"arguments"，{t}}}

{x, {{"function", Sin}, {{"arguments" ， x}}}, {{"function", 
   Power}, {{2 "arguments" ， x}}}, {{"function", 
   Times}, {{"arguments" ， x y}}}, {{"function", 
   Plus}, {{"arguments" ， x y}}}, {{"function", 
   g}, {{"arguments" ， x y}}}, {{"function", 
   h}, {{"arguments" ， x y z}}}, {{"function", 
   Cos}, {{"arguments" ， y}}}}

Try it online!

我使用带限制条件f=!=List的原因是：这样的话 整个列表将匹配，因为它的元素与模式_ _匹配 .

{x, Sin[x], x^2, x*y, x + y, g[y, x], h[x, y, z], Cos[y]} /. 
 f_[t__] -> {{"function", f}, {"arguments", {t}}}

{{"function", 
  List}, {"arguments", {x, Sin[x], x^2, x y, x + y, g[y, x], 
   h[x, y, z], Cos[y]}}}

Try it online!

我们稍后将讨论受限制的模式，因为现在只将它看作带有特定条件的模式。与此同时，这个示例本身表明，在定义模式时必须小心，因为在比我们预期的外的情况下可以匹配。

解决我们的问题

回到我们最初的问题，这里有一个解决方案:

{x, Sin[x], x^2, x*y, x + y, g[y, x], h[x, y, z], Cos[y]} /. 
 f_[t__] /; f =!= List :> (f[t] /. x -> 10)

{x, Sin[10], 100, 10 y, 10 + y, g[y, 10], h[10, y, z], Cos[y]}

Try it online!

解释：

step1：使用括号设置正确的优先级

keyboard_arrow_down

这个解决方案在几个方面都很出色。

首先，我们在整个规则中使用了一个外部规则，在括号中使用了内部规则。它的意思是:一旦我们发现任何与“第一”(或“外部”)规则的模式匹配的内容，就根据“第二”或“内部”规则立即替换该表达式。第一个(“外部”)规则在这里充当第二个(“内部”)规则的过滤器。如果我们省略括号，就得不到想要的结果:

{x, Sin[x], x^2, x*y, x + y, g[y, x], h[x, y, z], Cos[y]} /. 
  f_[t__] /; f =!= List :> f[t] /. x -> 10

{10, Sin[10], 100, 10 y, 10 + y, g[y, 10], h[10, y, z], Cos[y]}

Try it online!

发生这种情况的原因是：

规则应用程序是左关联的。因此，第一个规则首先被应用，并且基本上是空闲的，因为它本身只说f_[t__] /;f = != List:> f[t]，即用自身替换此形式的表达式

step1 = {x, Sin[x], x^2, x*y, x + y, g[y, x], h[x, y, z], Cos[y]} /. 
  f_[t__] /; f =!= List :> f[t]

{x, Sin[x], x^2, x y, x + y, g[y, x], h[x, y, z], Cos[y]}

第二个规则应用于step1 ，它说:当 x 发生时，用10 替换x 。因此，例如，列表的第一个元素，它只是 x ，不是任何函数的参数，也被替换了。

step1 /. x -> 10

{10, Sin[10], 100, 10 y, 10 + y, g[y, 10], h[10, y, z], Cos[y]}

但是，如果使用括号，则可以确保第二个规则实际应用于第一个规则“过滤”的结果

step2：使用RuleDelayed为内部规则提供正确的参数

keyboard_arrow_down

这里的第二个微妙之处是，我们使用RuleDelayed而不是Rule作为第一个(外部)规则。很容易看出，如果我们不这样做，就不会得到想要的结果

{x, Sin[x], x^2, x*y, x + y, g[y, x], h[x, y, z], Cos[y]} /. 
 f_[t__] /; f =!= List -> (f[t] /. x -> 10)

{x, Sin[x], x^2, x y, x + y, g[y, x], h[x, y, z], Cos[y]}

原因是在规则的情况下，在替换规则之前要对规则的r.h.s.进行评估。回顾Mathematica求值的一般策略，内部规则将在外部规则之前求值(这是括号所确保的)。但是到那时，外部规则中的 t 仍然与列表中匹配的部分没有任何关系，只是保持 t 。因此，规则f[t] /.x -> 10 完全空闲，计算结果为f[t]。这就产生了第一条规则f_[t__] /;f = != List ->f[t]，它也是空闲的。

step3,使用跟踪命令来确认

keyboard_arrow_down

{x, Sin[x], x^2, x*y, x + y, g[y, x], h[x, y, z], Cos[y]} /. 
  f_[t__] /; f =!= List -> (f[t] /. x -> 10) // Trace

{{{{x->10,x->10},f(t)/. x->10,f(t)},f_(t__)/;f=!=List->f(t),f_(t__)/;f=!=List->f(t)},{x,sin(x),x^2,x y,x+y,g(y,x),h(x,y,z),cos(y)}/. f_(t__)/;f=!=List->f(t),{List=!=List,False},{Sin=!=List,True},{Power=!=List,True},{Times=!=List,True},{Plus=!=List,True},{g=!=List,True},{h=!=List,True},{Cos=!=List,True},{x,sin(x),x^2,x y,x+y,g(y,x),h(x,y,z),cos(y)}}

Try it online!

如果我们使用RuleDelayed, 外部规则(包括内部规则)的rhs将不会被计算，直到为外部规则找到一些匹配。这将允许内部规则通过外部规则匹配来 操作 表达式 。

1.2.4.11 Rule 比对 RuleDelayed 何时使用他们两个

何时使用RuleDelayed更合适

一般来说，在规则的r.h.s.不确定的情况下，使用RuleDelayed通常更安全。

例如，如果模式定义标签(如fin f_ ， tint__等)具有任何全局值，我们刚才讨论的所有替换都将被完全破坏。例如，这是我们上面使用的规则之一，但是一些模式标签在应用规则之前有一些全局值:

f = "Global Function";
t = "Global Value";
{x, Sin[x], x^2, x*y, x + y, g[y, x], h[x, y, z], Cos[y]} /. 
 f_[t__] /; f =!= List -> {{"function", f}, {"arguments", {t}}}

{x,{{function,Global Function},{arguments,{Global Value}}},{{function,Global Function},{arguments,{Global Value}}},{{function,Global Function},{arguments,{Global Value}}},{{function,Global Function},{arguments,{Global Value}}},{{function,Global Function},{arguments,{Global Value}}},{{function,Global Function},{arguments,{Global Value}}},{{function,Global Function},{arguments,{Global Value}}}}

再说一次，这是因为在任何程序之前，规则的r.h.s.已经被评估过了。为了安全，使用规则延迟

f = "Global Function";
t = "Global Value";
{x, Sin[x], x^2, x*y, x + y, g[y, x], h[x, y, z], Cos[y]} /. 
 f_[t__] /; f =!= List :> {{"function", f}, {"arguments", {t}}}

{x,{{function,Sin},{arguments,{x}}},{{function,Power},{arguments,{x,2}}},{{function,Times},{arguments,{x,y}}},{{function,Plus},{arguments,{x,y}}},{{function,g},{arguments,{y,x}}},{{function,h},{arguments,{x,y,z}}},{{function,Cos},{arguments,{y}}}}

Try it online!

在第一种情况中，通过像上面那样编写模式，我们没有节省多少 代码
